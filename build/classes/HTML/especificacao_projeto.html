Introdução Este documento especifica os casos de uso e requisitos não-funcionais (RNFs) do projeto intitulado CWS - Manager, referente ao projeto final da disciplina de Projeto Integrador I (PI33A), contida no terceiro semestre da matriz curricular do curso de Tecnologia de Desenvolvimento de Sistemas para Internet, ministrada pela Profa. Aretha Barbosa Alencar. O objetivo do documento é apresentar as soluções elaboradas e discutidas durante o semestre para a construção da análise e do projeto de um software fundamentado nas idéias do paradigma orientado à objetos. 1.1 Descrição do sistema O CWS - Manager será capaz de gerenciar entrada e saída de veículos, controle de serviços realizados, o cadastro de clientes, funcionários, veículos, tipo de serviços de um lava-rápido. O Sistema também poderá emitir uma série de relatórios e contará com um agendamento de serviços online(WebSite), onde o cliente acompanhara os horários disponíveis e poderá agendar o serviço que será realizado em seu veículo. 1.2 Objetivo do sistema O principal objetivo do CWS - Manager é controlar os serviços realizados e o gerenciamento de clientes, funcionários, veículos de um lava-rápido. Ele também oferecerá mobilidade ao cliente, que poderá consultar e reservar seu serviço via WebSite. 1.3 Sistemas Similares - Sistema de gerenciamento de Lava-Rápido e Estacionamento Este software foi desenvolvido para controle de estacionamentos e lava-rápido, possui telas intuitivas, de fácil utilização. Imprime tickets de entrada de veículos e campos de cadastro como placa, nome do carro, código, data, chave, box, telefone do proprietário, convênio com estabelecimentos e muito mais. Fonte: http://ultradownloads.com.br/download/Gerenciamento-de-Lava-Rapido/#ixzz2WiHYI8Bk - Lava Rápido 3.0 É um prático sistema de gerenciamento para lava rápidos. Com ele é possível controlar a entrada de veículos, valores para todos eles e tipos de lavadas para cada veículo que entrou. Fonte: http://ultradownloads.com.br/download/Lava-Rapido/#ixzz2WiIOgVR3 1.4 Convenções, termos e abreviações A correta interpretação deste documento exige o conhecimento de algumas convenções, termos específicos e abreviações, que são descritos a seguir. - CWS – Manager: Car Wash System – Manager - UC: Casos de uso - RF: Requisitos funcionais - RNF: Requisitos não funcionais 1.4.1 Identificação dos casos de uso Os casos de uso devem ser identificados com um identificador único. A numeração inicia com o identificador [UC01] e prossegue sendo incrementada à medida que forem surgindo novos casos de uso. A nomenclatura dos fluxos secundários dos casos de uso é dada por uma sigla e por um número. A sigla deve ser FA para fluxos alternativos e FE para fluxos de erro. O número é um seqüencial que inicia de 01. Um exemplo de fluxo alternativo é [FA01] e um exemplo de fluxo de erro é [FE01]. O número do identificador reinicia a cada caso de uso. Para referenciar casos de uso em qualquer local do documento, o identificador do caso de uso é utilizado. Por exemplo, a pré-condição de um caso de uso poderia conter o seguinte texto: “Este caso de uso demanda que o usuário da aplicação esteja logado, como descrito em [UC12]”. Para referenciar um fluxo secundário fora do caso de uso que o define, é necessário utilizar o identificador do caso de uso concatenado com um ponto e com e o identificador do fluxo. Por exemplo, a descrição de um caso de uso poderia conter o seguinte texto: “Este caso de uso permite que o usuário edite informações avançadas do seu perfil e é disparado quando o usuário clica no botão Informações Avançadas durante seu cadastro, conforme descrito no fluxo alternativo [UC01].[FA02]”. 2. Lista de Requisitos Abaixo encontra-se a lista de requisitos do projeto. 1RF: O sistema deve permitir que os funcionários possam fazer o login no sistema, o login/senha são previamente definidos no seu cadastro; 2RF: O sistema deve gerenciar o cadastro dos clientes, contendo seus dados(nome, CPF, endereço, etc.), relacionar a um ou mais veículos, alem disso login/senha de acesso para o sistema web; 3RF: O sistema deve gerenciar o cadastro dos funcionários, contendo seus dados(nome, CPF, endereço, etc.) e login/senha para acesso do sistema; 4RF: O sistema deve gerenciar o cadastro dos veículos, contendo seus dados como modelo, cor, placa; 5RF: O sistema deve gerenciar o cadastro dos tipos de serviços prestados, como o preço e tempo para realizar o serviço; 6RF: O sistema deve abrir ordem de serviço, onde o funcionário informara o cliente, veículo e o tipo de serviço que será prestado, também conterá o status, data e hora de inicio; 7RF: O sistema deve fechar ordem de serviço, aonde o funcionário informara a data e hora de conclusão e o custo final do serviço; 8RF: O sistema deve emitir relatórios; 9RF: O sistema deve permitir que o Cliente faça login via Web; 10RF: O sistema deve permitir que o Cliente possa verificar os horários disponíveis para agendar um serviço através de um WebSite; 11RF: O sistema deve permitir que o Cliente agende um serviço através de um WebSite. 2.1 Matriz de Rastreabilidade 3. Casos de Uso Diagramas de caso de uso representam requisitos funcionais do sistema, provendo uma visualização sucinta do comportamento do mesmo. Normalmente a descrição do caso de uso é uma descrição na qual detalhes internos sobre como o sistema responde as ações de um ator estão ausentes ou descritas muito resumidamente, devido ao fato de os clientes não terem interesse em particularidades técnicas do sistema. A notação simplificada mostrada nos diagramas de casos de uso tem a finalidade de fornecer uma maneira intuitiva para pessoas que mesmo não familiarizadas com UML possam entender as funcionalidades do sistema. Um diagrama de casos de uso contém atores e casos de uso, onde eles se relacionam entre si, os relacionamentos podem ser ator com caso de uso, ator com ator em relacionamento de herança, caso de uso com caso de uso em relacionamento de herança, extend e include.
